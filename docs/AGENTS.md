# AGENTS.md

## Контекст проекта
Проект реализует внутреннее бронирование корпоративных ресурсов в рамках организации (tenant).
Важные бизнес-процессы системы:
- Управление организациями
- Управление пользователями и ролями
- Управление ресурсами
- Бронирование и использование ресурсов

## Границы проекта
В текущем проекте **не** реализуются:
- сложные процессы согласования между несколькими ролями;
- интеграции с внешними системами;
- дублирование заявок во внешние контуры;
- финансовый/бухгалтерский/эксплуатационный учет ресурсов.

Фокус: понятный базовый сценарий внутреннего бронирования.

## Ответственность этого репозитория (текущий модуль)
Явная зона ответственности в данном проекте: **Управление пользователями и ролями**.

Модуль покрывает:
- добавление пользователей в организацию;
- приглашение пользователей в организацию;
- назначение и изменение ролей;
- управление правами доступа на основе ролей;
- проверку принадлежности пользователя к организации;
- обеспечение tenant-изоляции данных при любых операциях с пользователями.

## Ключевые бизнес-правила
1. Пользователь может существовать без организации.
2. Любой пользователь может создать организацию.
3. При создании организации обязательно указывается ответственный пользователь (owner/admin).
4. Операции управления пользователями и ролями выполняются только в рамках конкретной организации.
5. Пользователь не может назначать права за пределами собственной организации.
6. Роль и права должны валидироваться на уровне домена и application-слоя.
7. Удаление/деактивация пользователя не должно нарушать целостность исторических данных.

## Минимальная ролевая модель (базовая)
- `Owner` — полный доступ в рамках организации, управление ролями и участниками.
- `Admin` — управление участниками и ролями (кроме ограничений, заданных политиками owner).
- `Member` — базовый доступ к пользовательским сценариям внутри организации.

> Примечание: детальные permission-матрицы можно расширять по мере развития требований.

## Сценарии модуля (MVP)
- Пригласить пользователя в организацию.
- Подтвердить приглашение и связать пользователя с организацией.
- Назначить/изменить роль пользователя.
- Получить список пользователей организации с их ролями.
- Удалить (или деактивировать) пользователя в рамках организации.

## Нефункциональные требования
- Tenant isolation by design: каждый запрос обязан учитывать `OrganizationId`.
- Аудит изменений ролей и членства (кто, когда, что изменил).
- Идемпотентность для повторных операций приглашения.
- Явная обработка ошибок доступа (`Forbidden`) и отсутствия сущностей (`NotFound`).

## Ожидаемые артефакты в коде
- Domain:
  - сущности: `OrganizationMembership`, `Role`, `Invitation` (или эквиваленты);
  - инварианты членства и ролей;
  - доменные события (опционально).
- Application:
  - use-cases/handlers для приглашения, назначения ролей, удаления участника;
  - DTO/команды/запросы;
  - policy/authorization checks.
- Infrastructure:
  - репозитории с фильтрацией по `OrganizationId`;
  - миграции и индексы для быстрого поиска по членству и ролям.
- API:
  - endpoint-ы для управления участниками и ролями;
  - валидация входа и корректные HTTP-коды.

## Definition of Done для задач модуля
Задача считается завершенной, если:
- реализован бизнес-сценарий в рамках организации;
- соблюдена tenant-изоляция;
- добавлены тесты (минимум unit + сценарный happy path);
- описаны ограничения и допущения;
- нет регрессий по правам доступа.

## Тестовые приоритеты
- Нельзя назначить роль пользователю из другой организации.
- Нельзя управлять участниками без достаточной роли.
- Повторное приглашение корректно обрабатывается.
- Деактивация пользователя блокирует дальнейшие операции от его имени в организации.
- Owner/Admin видят полный список участников, Member — согласно политике доступа.

## Принципы проектирования
- Явные границы контекстов (Organization vs Membership vs Booking).
- Минимизация связности с модулями ресурсов/бронирования.
- Расширяемость ролевой модели без массовых изменений API.
- Предсказуемые контракты ошибок и авторизации.
